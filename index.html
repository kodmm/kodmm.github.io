1. キャリアビジョン（2 行）と「理由」

人が壊れない社会を、仕組みとプロダクトで持続可能に実現するエンジニアになる

ドメインとアーキテクチャに沿ってプロダクトを正しく表現し、性能と運用まで含めて改善し続けるエンジニアになる

一文目は社会に対する北極星（ミス・孤独・認知負荷の前提）。事業が続かなければ届けられないので「持続可能」を同列に置く。二文目はそのために現場で担う職能定義で、設計・実装・性能・運用・観測まで一気通貫で責任を持ち、一文目を続く形で実装する、という順序。

3. 働く意義・価値観（全文構造化）

0. このドキュメントの目的
  • 自分の「働く理由」を言語化する
  • 会社選び・志望動機・面接でブレない軸を持つ

1. 働く意義（最上位）

根底にあるのは「困っている人・頑張っている人を IT の力で助けたい、支えたい、困りを軽減させたい」という一点。それを 3 つの柱に分解して働く理由としている。

  • テーゼ人が壊れない社会を作るために働く
  • ① ミスで壊れない人的ミスで自分を責めてしまう人を減らしたい。ミスを個人責任にせず仕組みで守り、自己肯定感を失ってしまうことがない世の中を目指す。「失敗してもシステムが受け止める」設計が理想。
  • ② 孤独で壊れない不安・孤独を減らし、感情面で人を守る。自己肯定感は「上げる」より「下げない」設計を優先する。
  • ③ 認知負荷で壊れないやりがいを感じにくいルーティン業務は IT で解決させ、その人が人・プロダクト・サービス、そしてプライベートの充実により多くの時間を使えるようにしたい。残業＝能力が発揮できない状態。疲労では品質が落ちる前提で設計する。
1-補. AI 時代における人間の役割

AI 活用が当たり前になったこの時代だからこそ、エンジニアリングで担う役割も変わる。加えて、人間の認知負荷が高い部分はシステムで軽減・削減したい・すべきと考えている（AI だけに任せず、仕組み全体で「頭を使わなくてよい領域」を減らす）。

  • IT に任せること反復・ルーティン・ヒューマンエラーが起きやすい定型業務。AI が代替できる「やりがいを感じにくい作業」はどんどん自動化する。
  • 認知負荷をシステムで下げる暗黙知の依存・手順の多さ・例外だらけの UI など、人の頭に無理に載せている負荷は、情報設計・自動化・ガードレール・ドキュメント・オンボーディングを含めてシステム側で吸収する。人が本当に判断すべきことに集中できるよう、エンジニアとして削るべきところを削る。
  • 人間が担うこと判断・共感・創造・関係構築など、人間にしか成せないことにより集中できる環境を IT で実現する。
  • 目指す世界「AI 活用のこの時代に、人間だけが成せることをより注力できる社会」。IT は人を代替するのではなく、人がより人間らしくいられるための基盤になるべきだと考えている。
2. 現実理解（転機）
  • 利益が出ない会社は続かない / キャッシュが尽きたら届けたいものも届けられなくなる
  • なくても困らないサービスは優先度が低く投資されにくい構造がある
  • 「良いこと」だけではダメ。「利益が出ること」だけでもダメ。統合条件は 良いこと × 利益が出ること
  • 利益は直接的な売上だけを指さない。プロダクトを通じたユーザーへの貢献度が積み重なり、継続利用・信頼・ブランドとして事業に還元されることも「利益が出ること」に含まれる。
3. 会社選びの必須 3 条件
  • 社会的必要性なくなると困る・日常に組み込まれている
  • ビジネスの持続性利益・投資・継続利用が見える
  • 人を守る設計思想ミス削減・認知負荷軽減・自己否定を生まない

この 3 つが揃う会社のみ選ぶ。

4. 価値提供の方向性
  • 機能的価値ミス削減・業務効率・正しくできる仕組み
  • 感情的価値不安・孤独の軽減。自己肯定感は「上げる」より「下げない」
  • 避ける設計過度な比較・拒絶の可視化・理由がわからない失敗
  • 目指す設計安心できる UI・失敗してもダメージが少ない・自分の価値を感じられる
5. 強みとの接続
  • 技術Go / TypeScript、フロント・バック両方
  • 経験toC での開発、UX 改善、レコメンド/検索
  • スクラム約 2.5 年スコープ調整・チーム運営・心理的安全性
  • 統合人を守るプロダクトとチームを作れるエンジニア
6. 働き方の定義
  • NG納期起因の残業、認知負荷が高い状態での作業、スコープ調整できない環境
  • OK熱中による自発的残業、スコープ調整可能、アジャイル / スクラム
  • その他条件週リモート 3-2、CI/CD、技術投資、年収 650〜700 万以上、心理的安全性
7. キャリア戦略と結論
  • 2 軸ミス削減（機能）と孤独解消（感情）。優先は必須サービス × ミス削減、サブは孤独解消
  • 一文人が壊れない社会を、持続可能なビジネスとして実現する
  • 分解人を守る（思想）／仕組みで解決する（エンジニアリング）／続く形にする（ビジネス）
  • 最後に優しさだけでは続かない。プロダクトへの貢献が利益として還元されない形では満たされない。両方やる。

5. 強みの整理

技術軸
  • PHP→Go の段階的マイグレーションを ratio-based で設計・運用（移行期の事故抑制と段階的検証を両立）
  • MySQL / Redis の実態調査（redis-cli, EXPLAIN）に基づく INDEX 設計と JOIN 最適化
  • Fastly VCL + Terraform でエッジ層の振り分けを設計（エリア別 / テーマ別 / 予約系）
  • クリーンアーキテクチャ（usecase / repository / interfaceadapter）を一貫して適用
  • gRPC / GraphQL / SSR Vue / PHP / Go に跨るフルスタック（1 リポで完結しない施策を貫通）
  • PR 出力量と速度: 1,095 PR / 約 3 年（直近月 20〜50 PR、99% マージ率・self-close 除く）
スクラム / チーム運営
  • スプリントレビューアジェンダ運営を 3 年弱継続（再現可能な仕組み化）
  • 「振り返り共有会」を通じた横断的なチーム改善のハブ役
  • BAD/GOOD と議論セクションを定型化し心理的安全性を担保
  • 新メンバー向け「環境一覧」「DB 規約」等を整備し、オンボードを資産化
  • ポストモーテム / 障害対応フローへ継続的にコミット（運用とプロセスの一体運営）
いちエンジニアとしての立ち回り
  • 1,822 件のレビュー（コメント 1,391 件）— チームの知識循環の実質的中核
  • レビュートーンが柔らかく対立を避ける（「優先度は低いくらいの FB ✋」「余力あれば 👌」）
  • セルフレビューを高頻度で残し、レビュアー負荷を下げる（[memo] / 要確認でファクトを先出し）
  • 歴史的経緯（8 年前の Redis 投入）まで Slack / AI / コードを横断検索して根拠化
  • 意思決定が数値ベース（≥100ms で INDEX、画像で前後比較）
  • クロスリポジトリ施策の整合性（PHP↔Go 同等動作、CDN↔Frontend の 3 PR 分割）
  • AI ツール（Cursor / Devin）を活用しつつ自己検証を欠かさない
  • 新しい技術を積極的に採用するのが好き。grid / flex のように同じ出面でも手段が異なる場合は、変更容易性・可読性・その UI 表現に適した手段を優先して選定する
  • どちらが適切か判断できない場合は、新しい方を採用して Try & Error で課題を発見する機会とする姿勢を持つ

外部決済残高 API → API Gateway → Webhook Lambda が大量の同期リクエストでライター DB 最大接続数を超過。全ページが 500 を返す中、原因特定 → 対象ユーザー洗い出し → 同期クエリ作成・実行 → Playbook 整備 → システム改善 Issue 化を 1 人で実施。Action Items の 4 件（investigate / repair / detect / prevent）すべて自分が owner になり DONE。

  • インフラ: 障害対応 Playbook 整備
  • IaC: Webhook 基盤 Terraform 化

学び: 「対応のほとんどは 1 人で行っていたので、oncaller やチームを巻き込んで複数人で対応した方が良さそう」と自己反省を文書化 — 単独対応の限界を認識し、巻き込み設計を次の課題として明示。

キャッシュレイテンシ悪化障害（2026-04）周辺コードレビュー / 設計波及修正

  • バックエンドサービス: fncache レースコンディション修正
  • インフラ IaC: 最低タスク数 / キャッシュ監視整備
  • ポストモーテム（reviewer として参加）

学び: 障害契機にキャッシュ処理の「不要な KEYS/Exists」「cache-in-cache 破損」「keyspace-events 未設定」まで波及修正 — 単一障害から設計負債を洗い出す目利き。

OnCaller 運用ドキュメント整備（2025-09）ドキュメント整備リード

「OnCaller」ページ（v24）の最終編集者として、期待役割の言語化・APM 登録 / AWS 権限申請フロー・hotfix マージ権限・関連チャンネルを一覧整備。

学び: 「正式に定められた申請フローはない」と書き残し pending 化 — 暗黙運用を可視化することでプラットフォームチームへの巻き取りを促す動き方。

強み
  • 検知 → 暫定復旧 → 恒久対応 → Postmortem → Playbook → システム改善を一気通貫で完遂できる
  • Lambda × API Gateway × Writer DB × APM × CDN の多コンポーネント障害でも自律的に動ける守備範囲
  • 事後に必ず Action Item を 4 種（investigate / repair / detect / prevent）で構造化
  • OnCaller 運用そのものをドキュメント整備の対象として扱い属人化を解消する方向に動かす
  • 障害から「周辺の設計問題」までスコープを広げて再発防止できる深さ
  • 決済 / Lambda / DB / フロント / CDN を跨ぐインシデント切り分け力
  • エラーハンドリング・ログ出力は「エラーを見つけやすい仕組み」を第一に設計。適切なエラーメッセージ・ステータスコードを意識して実装する
  • OnCaller として障害対応時の緊張・焦りを自覚しているからこそ、わかりやすいエラーログを実装時から仕込み、自他ともに早急解決できる状態を作ることを習慣にしている
要言語化
  • 「1 人で完了させる」傾向が自覚されている — 巻き込み・分散対応の設計は次の課題
  • primary 担当回数 / MTTR などのポートフォリオは可視化できていない
  • OnCaller 運用の根本改善（申請フロー正式化）は pending 状態
  • グループ外領域の障害対応経験量は本人の目線では分かりづらい

Q1. なぜ転職しようと思ったか？

社内で会社の利益起因でレイオフ対象になったことが発端です。残っても良いと言われたものの、話が来ている以上、今後の会社のビジョンに対して自分が貢献できる可能性は低いと判断されたと受け取りました。

元々、入社から約 2 ヶ月の頃に出向の話がありました。そのときは社会人になってすぐで自分に自信がなかったこと、何よりこの会社で成し遂げたいことが何一つ叶えられていない時期だったため、会社に残ることを決断しました。

今回を含め、似たような機会が計 2 回あったため、これを機に「今後、自分が IT を活用して解決したいこと」を考え直し、環境を変えることを決断しました。

Q2. テストはどう書いているか？どんな方針で？

フロントエンドに限らず、バックエンド・インフラ周りも、最低でもドメインロジックや施策に関する仕様部分には必ずテストコードを入れ、品質を担保するようにしていました。DB 周りは基本 mock を活用していました。

また、複雑な部分や再利用可能な部分もテストで品質を担保するようにしていました。

Q3. 最もチャレンジングだった技術的課題は？

主要ページの最重要ロジックをレガシーな PHP から Go へリプレイスしたことです。

移行の設計をはじめ、移行したロジック自体が等しいかを検証するマイグレーションテスト、さらには移行による不要な仕様の整理やパフォーマンスをより良くするための改善、お店さんからお金をいただいて掲載順位が変わるという移行リスクが高い環境下での段階的移行、メトリクスやパフォーマンスの監視を、すべて主体的に自ら進んで全うしました。

設計においては、責務やアーキテクチャに沿った形・移行難易度・スコープなどを考慮して、その時点で最善の設計を提案しました。今後の DevOps にも影響する点があったため、その整備も併せて行いました。

不要な部分や不要に cache から取得している部分はハードコーディングし、ホワイト化・パフォーマンス改善・依存部分の除去も行いました。パフォーマンスについては、処理が重い部分の予測や dev 環境での実測をもとに「ボトルネックを探して対応」の反復を続け、改善しました。具体的には INDEX を貼ったり、Redis の cache に載せるなどの対応を行いました。

段階的移行の手段としては CDN のパーセンテージルーティングを駆使し、すぐ戻せる準備や影響の少ない箇所から移行することを提案し、実現しました。

Q6. 最近キャッチアップしている技術・トレンドは？

① AI を使った開発フローの実例

AI を活用して施策の起案からリリースまでを一気通貫で回している事例を積極的にキャッチアップしています。AI でできることが急速に広がる中で、「人間としての知恵の使い所はどこか」という問いの答えを追い続けています。なお、AI は世界中の知恵の集積・宝庫だと捉えており、うまく活用することで自分自身の思考の質も高められると考えています。

② 今後も変わらない基礎領域

時代が変わっても必要とされ続けるスキルとして、DB 設計・クエリ最適化の知見と、フロントエンド / BFF / バックエンドを分離したマルチリポジトリ構成のアーキテクチャを継続的に学んでいます。「人間にわかりやすいコードは AI にもわかりやすい」という考え方を軸にしており、可読性・変更容易性の高い設計は AI コストの観点でも投資対効果が高いと判断しています。会社・自分のキャリア双方にとって不可欠なスキルと位置づけています。

③ 現在の技術スタックをより深く活かす

現職で採用している技術（Go / Vue・Nuxt 3 / gRPC / Terraform / Fastly / Datadog など）について、より効果的な使い方や最新のベストプラクティスを常に追いかけています。導入済みの技術の「使いこなし度を上げる」ことが、チームの生産性向上と障害対応力の強化に直結すると考えているためです。

Q7. CI/CD はどう構築・運用しているか？

PR 時に GitHub Actions で lint / format / test を自動実行しています。Terraform 側は PR チェックで plan を実行し、main マージ後に apply を自動化しています。リリースはタグを打つ形をすでに入社当初から採用しており、大きな課題感はなかったため、そのフローに倣っていないリポジトリがあれば積極的に統一する変更を加えてきました。

自主的な CD 整備としては、レガシーなリポジトリを入社当初から施策や手隙のタイミングでリプレイスする動きを続けていました。レガシー依存な開発が減少し、依存するページや API も少なくなってきたタイミングで、リリースフローを「branch hook による 1 日 1 回の定時リリース」から「いつでもタグを打てばリリースできる仕組み」に変更しました。

この変更によって得られた効果は 3 点あります。①社内リポジトリの CD フロー統一化の向上、②リリース担当チームの廃止（担当者がリリースを失念する人的ミスが発生していたが、PR マージ者が自主的にリリースできるようになり漏れがゼロに）、③後から加わるメンバーの認知負荷の軽減です。

Q5. 失敗・反省した経験は？

① OnCaller 中、人を巻き込むのが遅れた経験

弊社では 1 週間単位で primary / secondary の 2 名体制でサービスの障害対応を 24 時間行っています。基本的に primary の対応が困難な場合は secondary へエスカレーションするフローです。

あるインシデントでは、原因追及と対応に夢中になっていたことに加え、業務時間外だったため secondary の方の大切なプライベートな時間を奪ってしまう申し訳なさが拭えず、エスカレーションが遅れました。結果として障害時間が超過し、ユーザーさんにご迷惑をおかけしてしまいました。

たまたま secondary の方が自ら障害に気づいてサポートしてくれ、プライベートな時間にも関わらず数名のエンジニアが駆けつけてくれたことで事なきを得ました。しかし primary にも関わらず、人が来てくれた際にインシデントコマンダーになれず、他者に委ねてしまいました。

この経験から、15 分以上解決の見込みがない場合はサービスの安定稼働を最優先に secondary へエスカレーションするフローを OnCaller 内で確立しました。

② 入社当初、設計確認を怠り PR で 100 件以上の議論を発生させた経験

入社当初、やる気に満ち溢れていたあまり、自分のアピールの仕方を履き違え、設計の確認やコーディングルールを適切に把握しないまま実装を進め、PR 内で 100 件以上の議論を発生させ、リリースを遅らせてしまいました。

この経験から、不明瞭で解像度の低い部分は事前に高める動き（認識合わせ・設計段階で Slack 等に設計案を投稿して FB をもらう）を徹底するようになりました。結果として手戻りの少ない開発ができるようになりました。

Q4. 仕様が曖昧なまま実装しなければならないとき、どうするか？

まず施策担当者と仕様の認識を揃える時間を取ります。議論の末すぐに決まらない場合は、すでに決まっている部分や戻る影響が少ない部分から実装を始めます。それも厳しい場合は、無理に進めず待つ姿勢を取ります。

スクラムの環境下だったこともあり、早く出して早く PDCA を回すことを念頭に置きながら、その時点で取れる最善の方法を選ぶようにしていました。
